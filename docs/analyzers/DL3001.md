# DL3001 - Use named arguments

Requires named arguments when the argument's intent is not self-evident at the call site.

| Rule   | Category | Default Severity | Code Fix |
|--------|----------|-----------------|----------|
| DL3001 | Style    | Warning         | Yes      |

## Why

Positional arguments for literals and mismatched variable names make call sites ambiguous:

```csharp
// What do these arguments mean?
CreateUser("Alice", true, 30, null);
```

Named arguments make intent explicit:

```csharp
CreateUser(name: "Alice", isAdmin: true, age: 30, department: null);
```

## When the diagnostic fires

DL3001 is reported when **any** of these conditions are true:

### 1. Ambiguous literals

The argument is a literal whose meaning is not obvious from context:

- `null`
- `true`, `false`
- Numeric literals (`42`, `0`, `-1`)
- String literals (`"hello"`)
- Character literals (`'x'`)
- `default` and `default(T)`

```csharp
Retry(3, true); // DL3001 on both arguments
Retry(retries: 3, exponentialBackoff: true); // OK
```

### 2. Variable name mismatch

The argument is a variable or member access whose name does not match the parameter name (case-insensitive, ignoring leading underscores):

```csharp
var total = 5;
SetCount(total); // DL3001 — 'total' != 'count'
SetCount(count: total); // OK

var count = 5;
SetCount(count); // OK — names match

var _count = 5;
SetCount(_count); // OK — underscore stripped, matches 'count'
```

### 3. Expressions where intent is unclear

Method call results, complex expressions, and other non-identifier arguments require named arguments:

```csharp
Foo(GetCount()); // DL3001 — intent not clear
Foo(count: GetCount()); // OK

Foo(a + b); // DL3001
Foo(count: a + b); // OK
```

### 4. [NamedArgument] attribute

The parameter is marked with `[NamedArgument]`, requiring named arguments **always** — even when the variable name matches the parameter name:

```csharp
using DemoriLabs.Diagnostics.Attributes;

public void Send([NamedArgument] string to, [NamedArgument] string body) { }

var to = "alice@example.com";
var body = "Hello";
Send(to, body); // DL3001 on both — name match is not enough
Send(to: to, body: body); // OK
```

## When the diagnostic does NOT fire

- Argument is already named (`count: 42`)
- Argument has `ref`, `out`, or `in` keyword
- Argument is in an indexer (`list[0]`)
- Parameter is `params`
- Argument is a variable or member access matching the parameter name

## Examples

### Violation

```csharp
public void Configure(int timeout, bool retry, string endpoint) { }

// DL3001 on all three arguments
Configure(30, true, "https://api.example.com");
```

### Fixed

```csharp
Configure(timeout: 30, retry: true, endpoint: "https://api.example.com");
```

### Constructor and base calls

Named arguments are also required in constructor calls:

```csharp
var person = new Person("Alice", 30); // DL3001
var person = new Person(name: "Alice", age: 30); // OK
```

## Code fix

**Add argument name '{name}:'** — inserts the parameter name before the argument value.
