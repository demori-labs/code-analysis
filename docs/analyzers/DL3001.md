# DL3001 - Use named arguments

Requires named arguments when a method has many parameters, or when a parameter is explicitly marked with `[NamedArgument]`.

| Rule   | Category | Default Severity | Code Fix |
| ------ | -------- | ---------------- | -------- |
| DL3001 | Style    | Warning          | Yes      |

## Why

Positional arguments become hard to follow as parameter count grows:

```csharp
// What do these arguments mean?
CreateUser("Alice", true, 30, null);
```

Named arguments make intent explicit:

```csharp
CreateUser(name: "Alice", isAdmin: true, age: 30, department: null);
```

## When the diagnostic fires

DL3001 is reported when **either** of these conditions is true:

### 1. Parameter count exceeds the threshold

When a method has more **visible** parameters than the configured threshold (default: 2), all unnamed arguments whose name does not match the parameter are reported.

**Visible parameters** exclude:

- `this` (extension methods)
- `params`
- `ref`, `out`, `in`
- Optional parameters (those with default values)

Optional parameters are excluded from the count, but if explicitly passed on a method already above threshold, their arguments are still checked for name matching.

**Name matching** is case-insensitive, ignoring leading underscores. Additionally, an argument name matches if the parameter name is the argument name followed by the parameter's type name (e.g. `options` matches `optionsAction` when the type is `Action<T>`).

```csharp
Foo(n, a, e);
// Foo(string name, int age, string email) → 3 visible > threshold of 2 → DL3001

Foo(name, age, email); // OK — names match

Foo(name, years, email); // DL3001 on 'years' only

// Extension method — 'this' not counted, params not counted
logger.LogDebug("Message: {Id}", id); // OK — 1 visible parameter

// Optional parameters not counted
services.AddDbContext<MyDb>(options => { }); // OK — 1 visible parameter

// Type-suffix matching
Action<DbOptions> options = ...;
Foo(options, name, count);
// Foo(Action<DbOptions> optionsAction, ...) → 'options' matches 'optionsAction'
```

### 2. [NamedArgument] attribute

The parameter is marked with `[NamedArgument]`, requiring named arguments **always** — regardless of threshold or name matching:

```csharp
using DemoriLabs.CodeAnalysis.Attributes;

public void Send([NamedArgument] string to, [NamedArgument] string body) { }

var to = "alice@example.com";
var body = "Hello";
Send(to, body); // DL3001 on both — name match is not enough
Send(to: to, body: body); // OK
```

## When the diagnostic does NOT fire

- Argument is already named (`count: 42`)
- Argument has `ref`, `out`, or `in` keyword
- Argument is in an indexer (`list[0]`)
- Parameter is `params`
- Method has visible parameters at or below the threshold (and no `[NamedArgument]`)
- Argument variable/member name matches the parameter name (above threshold)
- Argument name matches the parameter name with a type suffix (e.g. `options` → `optionsAction`)
- Argument is inside an expression tree (`Expression<Func<...>>`)

## Examples

### Violation

```csharp
void Configure(int timeout, bool retry, string endpoint) { }

Configure(30, true, "https://api.example.com"); // DL3001 on all three
Configure(timeout: 30, retry: true, endpoint: "https://api.example.com"); // OK
```

### Below threshold

```csharp
Foo(true);
Foo("hello");
throw new Exception($"Operator '{op}' not supported.");
```

### Name matching

```csharp
void Foo(string name, int age, string email) { }

Foo(name, age, email);   // OK — names match
Foo(name, years, email); // DL3001 on 'years' only
Foo(_name, _age, _email); // OK — leading underscores stripped
Foo(o.Name, o.Age, o.Email); // OK — member name compared
```

### Type-suffix matching

```csharp
void Register(Action<DbOptions> optionsAction, Func<int> countFunc, string name) { }

Action<DbOptions> options = ...;
Func<int> count = ...;
Register(options, count, name); // OK — 'options' matches 'optionsAction' (suffix = type Action)
```

### Extension methods

The `this` parameter is excluded from the visible count:

```csharp
// ILogger.LogDebug(this ILogger, string?, params object?[])
// visible params: 1 (message only — this and params excluded)
logger.LogDebug("Start processing: {Id}", id); // OK
```

### Optional parameters

Parameters with default values are excluded from the visible count:

```csharp
// AddDbContext<T>(this IServiceCollection, Action<DbContextOptionsBuilder>? = null, ServiceLifetime = Scoped, ServiceLifetime = Scoped)
// visible params: 0 (this excluded, all others optional)
services.AddDbContext<MyDb>(options => options.UseNpgsql(conn)); // OK
```

If explicitly passed on a method already above threshold, optional arguments are still checked:

```csharp
void Foo(string name, int age, string email, int limit = 10) { }

Foo("Alice", 30, "a@b.com", 5); // DL3001 on all four (3 visible > threshold, limit also checked)
```

### Constructor calls

```csharp
var person = new Person("Alice", 30, "a@b.com"); // DL3001
var person = new Person(name: "Alice", age: 30, email: "a@b.com"); // OK
```

### Expression trees

Arguments inside expression trees are suppressed (named arguments are invalid in expression trees):

```csharp
Expression<Func<bool>> expr = () => Foo(true); // OK — inside expression tree
```

## Code fix

**Add argument name '{name}:'** — inserts the parameter name before the argument value.

## Configuration

```ini
# Set the parameter count threshold (default: 2, so 3+ params trigger)
dotnet_diagnostic.DL3001.named_arguments_threshold = 3

# Adjust severity
dotnet_diagnostic.DL3001.severity = error

# Disable entirely
dotnet_diagnostic.DL3001.severity = none
```
