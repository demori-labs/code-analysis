# DL3005 - Use logical pattern

Detects chained comparisons, pattern checks, and `Nullable<T>.HasValue` / `.Value` accessors on the same variable that can be replaced with a single logical pattern (`is X or Y`, `is >= A and < B`, `is not X and not Y`).

| Rule   | Category | Default Severity | Code Fix |
| ------ | -------- | ---------------- | -------- |
| DL3005 | Style    | Warning          | Yes      |

## Priority over DL3003 and DL3004

DL3005 asserts the highest priority among the pattern matching analysers. When DL3005 resolves an `||`/`&&` chain, it deliberately suppresses the evaluation of individual leaf nodes. Analysers such as DL3003 (constant pattern) and DL3004 (negation pattern) are designed to detect if their target node belongs to a chain handled by DL3005, deferring to it immediately. This strict hierarchy prevents overlapping diagnostics and eliminates redundant noise.

```
DL3005 (logical chains)  ← DL3003 and DL3004 defer to this
DL3003 (single comparison, or outer wrapped expression) ← inner DL3003 defers to outer DL3003/DL3004
DL3004 (negation)        ← independent
```

## Rationale

Evaluating multiple constraints against a single variable using chained logical operators (`||`, `&&`) introduces unnecessary repetition and visual clutter. Constantly repeating the variable name forces the reader to parse redundant syntax, which obscures the fundamental intent: asserting a boundary or a set of valid states for a single piece of data.

Logical patterns consolidate these disparate checks into a unified, cohesive construct. For instance, `x is 1 or 2 or 3` or `x is >= 0 and < 100` firmly binds a declarative set of constraints directly to the target variable. This simple shift drastically reduces structural boilerplate, optimises the expression for linear left-to-right reading, and ensures the code remains focused on the actual domain logic.

## When the diagnostic fires

### `||` chains (or patterns)

- `x == 1 || x == 2 || x == 3` — equality chain
- `id == null || id == 0` — nullable null-or-value
- `id == null || id > 10` — mixed equality and relational
- `!id.HasValue || id == 0` — `Nullable<T>.HasValue` negation treated as `== null`
- `id is null || id is 0` — already pattern matching but separate `is` expressions

### `&&` chains (and patterns)

- `x != 1 && x != 2` — not-equals chain
- `x >= 0 && x < 100` — relational range
- `id != null && id > 0` — mixed not-equals and relational
- `id.HasValue && id != 0` — `Nullable<T>.HasValue` treated as `!= null`
- `id != null && id.Value > 0` — `Nullable<T>.Value` accessor recognised as the same variable
- `id is not null && id is not 0` — separate `is` expressions merged
- `id is not null && id is > 0` — mixed `is` patterns merged
- `x >= 0 && x < 100 && x != 50` — three or more leaves

All comparisons must reference the exact same symbol and compare against compile-time constants.

### `Nullable<T>` support

- `HasValue` and `!HasValue` are explicitly recognised as null/not-null checks on the underlying nullable variable.
- The `.Value` accessor is resolved back to the underlying nullable variable (e.g. `id.Value > 0` successfully matches with `id != null`).
- A strict semantic check guarantees this behaviour only applies to `System.Nullable<T>`. Custom types possessing `HasValue` or `Value` properties are deliberately ignored to prevent unsafe assumptions.

## When the diagnostic does NOT fire

- Different variables on each side (e.g. `x == 1 || y == 2`)
- Non-constant right-hand sides (e.g. `x == a || x == b`)
- String literal comparisons (e.g. `s == "a" || s == "b"`) — handled by dedicated string analysers
- Inside `Expression<Func<>>` lambdas (pattern matching is not supported in expression trees)
- Already using combined pattern matching (e.g. `x is 1 or 2 or 3`)
- Custom types with a `HasValue` property (not `Nullable<T>`)
- Custom types with a `Value` property (not `Nullable<T>`)

## Examples

### Violation

```csharp
// || chains
if (x == 1 || x == 2 || x == 3) { }    // DL3005
if (id == null || id == 0) { }          // DL3005
if (id == null || id > 10) { }          // DL3005
if (!id.HasValue || id == 0) { }        // DL3005
if (id is null || id is 0) { }          // DL3005

// && chains
if (x >= 0 && x < 100) { }             // DL3005
if (x != 1 && x != 2) { }              // DL3005
if (id != null && id > 0) { }          // DL3005
if (id.HasValue && id != 0) { }        // DL3005
if (id != null && id.Value > 0) { }    // DL3005
if (id is not null && id is not 0) { } // DL3005
if (id is not null && id is > 0) { }   // DL3005
```

### Fixed

```csharp
// || chains
if (x is 1 or 2 or 3) { }
if (id is null or 0) { }
if (id is null or > 10) { }
if (id is null or 0) { }
if (id is null or 0) { }

// && chains
if (x is >= 0 and < 100) { }
if (x is not 1 and not 2) { }
if (id is not null and > 0) { }
if (id is not null and not 0) { }
if (id is not null and > 0) { }
if (id is not null and not 0) { }
if (id is not null and > 0) { }
```

### No diagnostic

```csharp
// Already using combined pattern matching
if (x is 1 or 2 or 3) { }

// Different variables
if (x == 1 || y == 2) { }

// Non-constant
if (x == a || x == b) { }

// Custom type with HasValue — not Nullable<T>
public class Option { public bool HasValue { get; set; } }
if (!opt.HasValue || opt == null) { }
```

## Code fix

Rewrites the entire `||`/`&&` chain as a single, consolidated `is` pattern expression:

- `||` with `==` → `is X or Y or Z`
- `||` with mixed `==`/relational → `is X or > Y`
- `&&` with `!=` → `is not X and not Y`
- `&&` with relational → `is >= A and < B`
- `&&` with mixed `!=`/relational → `is not X and > Y`
- `HasValue`/`.Value` accessors are normalised to the base variable automatically.
