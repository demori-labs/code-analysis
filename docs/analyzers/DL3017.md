# DL3017 - Use string.Equals with StringComparison

Flags string comparisons using `==`, `!=`, `is "literal"`, or `is not "literal"` where no explicit `StringComparison` is specified.

| Rule   | Category | Default Severity | Code Fix |
| ------ | -------- | ---------------- | -------- |
| DL3017 | Style    | Warning          | Yes      |

## Why

The `==` and `!=` operators on strings compile to `string.op_Equality` / `string.op_Inequality`, which perform an ordinal comparison without making that intent explicit. Similarly, `is "literal"` and `is not "literal"` patterns compile to the same `op_Equality` call. Microsoft's [best practices for string comparison](https://learn.microsoft.com/en-us/dotnet/standard/base-types/best-practices-strings) recommend using `string.Equals` with an explicit `StringComparison` parameter to make the comparison semantics clear and avoid subtle culture-related bugs.

## When the diagnostic fires

- `str == "value"` or `"value" == str`
- `str != "value"`
- `str1 == str2` or `str1 != str2` (two string-typed expressions)
- `str is "value"`
- `str is not "value"`

## When the diagnostic does NOT fire

- Comparisons against `null` or `default` (those are null checks, not string comparisons)
- Both sides are compile-time constants (the compiler folds these)
- Inside expression trees (`Expression<T>` lambdas) where `StringComparison` overloads may not translate to SQL
- Complex patterns like `str is "a" or "b"` (only simple constant patterns are flagged)
- Non-string types (e.g. `int == 42`, `object == object`)

## Examples

### Violation

```csharp
public bool IsAdmin(string role)
{
    return role == "admin"; // DL3017
}

public bool IsGuest(string role)
{
    return role is "guest"; // DL3017
}

public bool IsNotRoot(string user)
{
    return user != "root"; // DL3017
}
```

### Fixed

```csharp
public bool IsAdmin(string role)
{
    return string.Equals(role, "admin", StringComparison.Ordinal);
}

public bool IsGuest(string role)
{
    return string.Equals(role, "guest", StringComparison.Ordinal);
}

public bool IsNotRoot(string user)
{
    return string.Equals(user, "root", StringComparison.Ordinal) is false;
}
```

### No diagnostic

```csharp
// Null checks are fine
if (name == null) { }
if (name is null) { }

// Already using string.Equals with StringComparison
if (string.Equals(name, "admin", StringComparison.OrdinalIgnoreCase)) { }

// Compile-time constants
if ("a" == "b") { }

// Complex patterns (out of scope)
if (name is "a" or "b") { }

// Inside expression trees
IQueryable<User> query = users.Where(u => u.Name == "admin");
```

## Related Microsoft rules

DL3017 covers operators (`==`, `!=`, `is`, `is not`) on strings. For **method calls** without an explicit `StringComparison`, enable the following Microsoft rules — they are disabled by default:

| Rule | What it flags |
| ---- | ------------- |
| [CA1307](https://learn.microsoft.com/en-us/dotnet/fundamentals/code-analysis/quality-rules/ca1307) | Any string method (`Equals`, `Compare`, `IndexOf`, `StartsWith`, etc.) that has a `StringComparison` overload but doesn't use it |
| [CA1309](https://learn.microsoft.com/en-us/dotnet/fundamentals/code-analysis/quality-rules/ca1309) | String comparison using non-ordinal `StringComparison` (e.g. `InvariantCulture`) |
| [CA1310](https://learn.microsoft.com/en-us/dotnet/fundamentals/code-analysis/quality-rules/ca1310) | String method that defaults to culture-specific comparison (`Compare`, `StartsWith`, `IndexOf`) |

To enable them, add to your `.editorconfig` or project file:

```ini
dotnet_diagnostic.CA1307.severity = warning
dotnet_diagnostic.CA1309.severity = warning
dotnet_diagnostic.CA1310.severity = warning
```

## Code fix

Replaces the comparison with `string.Equals(left, right, StringComparison.Ordinal)`. For inequality (`!=`) and `is not "literal"`, the replacement is wrapped with `is false`. Adds `using System;` if not already present. The original argument order is preserved.
