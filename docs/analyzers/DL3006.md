# DL3006 - Use declaration pattern instead of type check and cast

Detects `if (x is T)` followed by a cast `(T)x` in the if-body, which should use a declaration pattern `if (x is T t)` instead.

| Rule   | Category | Default Severity | Code Fix |
| ------ | -------- | ---------------- | -------- |
| DL3006 | Style    | Warning          | Yes      |

## Rationale

Separating a type check from its subsequent cast introduces an unnecessary mechanical redundancy. When you construct an `if (x is T)` guard clause and then manually cast `(T)x` within the block, you instruct the runtime to evaluate the object's type hierarchy twice.

More critically, you are structurally separating the proof of safety (the type check) from the consumption of that safety (the cast). This leaves a gap where maintenance errors can easily occur—for example, updating the type in the guard clause but forgetting to update the subsequent cast.

The declaration pattern (`is T t`) resolves this by unifying the test and the binding into a single, atomic operation. It securely binds the correctly typed reference only when the constraint is strictly satisfied, perfectly aligning the program's logic with its lexical scope. This leverages standard, out-of-the-box C# capabilities to eliminate boilerplate, prevent out-of-sync casts, and deliver a more straightforward, structurally sound implementation.

## When the diagnostic fires

- `if (x is T)` where the if-body contains one or more `(T)x` casts of the exact same expression to the exact same type.
- The `is` expression is located within the condition of an `if` statement (or forms part of a logical `&&` condition leading into the block).

## When the diagnostic does NOT fire

- `x is T` is used outside an if-condition (e.g., within a return expression or ternary operator).
- No matching explicit casts are found within the if-body.
- The cast targets a different type than the one verified in the type check.
- Inside `Expression<Func<>>` lambdas (pattern matching is not supported in expression trees).
- The code is already utilising a declaration pattern (`x is T t`).

## Examples

### Violation

```csharp
if (o is Animal)              // DL3006
{
    var a = (Animal)o;
    Console.WriteLine(a.Name);
}
```

### Fixed

```csharp
if (o is Animal animal)
{
    var a = animal;
    Console.WriteLine(a.Name);
}
```

### No diagnostic

```csharp
// Already using declaration pattern
if (o is Animal animal)
{
    Console.WriteLine(animal.Name);
}

// No cast in the body
if (o is Animal)
{
    Console.WriteLine("it's an animal");
}
```

## Code fix

Replaces the isolated type check `x is T` with the declaration pattern `x is T varName` within the condition. It then systematically replaces all matching `(T)x` casts in the corresponding block with `varName`. The variable name is deterministically derived from the type name, formatted in camelCase. Any collisions with reserved C# keywords are safely prefixed with `@`.
