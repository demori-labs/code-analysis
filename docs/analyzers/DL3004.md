# DL3004 - Use 'is false' or 'is not' instead of '!'

Detects use of the `!` operator for boolean negation and negated type/pattern checks, suggesting `is false` or `is not` instead.

| Rule   | Category | Default Severity | Code Fix |
| ------ | -------- | ---------------- | -------- |
| DL3004 | Style    | Warning          | Yes      |

## Rationale

The primary drawback of the logical negation operator (`!`) is its visual subtlety. Structurally, it is merely a thin vertical stroke with a dot. During rapid code scanning, it is notoriously easy to overlook. Furthermore, in many common coding typefaces, the `!` character shares a striking visual resemblance to `l`, `i`, `1`, or `|`.

Consider the visual density of expressions like `!isValid` or `!list.Any()`. The negation—the most critical part of the logical branch—relies entirely on a single pixel-width stroke that often blends into the adjacent ascenders. This visual ambiguity forces the reader to expend unnecessary cognitive effort parsing syntax rather than comprehending the underlying logic.

By contrast, `is false` is unambiguous. It transforms a subtle visual cue into an explicit semantic statement that reads naturally left-to-right. It clearly anchors the negated condition at the end of the expression. Similarly, wrapping pattern matches in negations (e.g., `!(x is string)`) requires the reader to parse from the inside out and track parenthesis boundaries. The construct `x is not string` is linear, direct, and entirely removes this cognitive overhead. The trade-off is slightly more keystrokes in exchange for absolute clarity.

## When the diagnostic fires

- `!flag` — any boolean negation
- `!string.IsNullOrEmpty(s)` — negated method calls returning bool
- `!(x is T)` — negated type check
- `!(x is null)` — negated null pattern
- `!(x is T t)` — negated declaration pattern
- `!(x is > 5)` — negated relational pattern

## When the diagnostic does NOT fire

- Operand is `bool?` (different semantics — `!null` is `null`, not `false`)
- Operand type has an overloaded `operator !`
- `!(a && b)` or `!(a || b)` — De Morgan's rewrite is a distinct concern
- Inside `Expression<Func<>>` lambdas (pattern matching is not supported in expression trees)
- The `!` expression is part of a `||`/`&&` chain that [DL3005](DL3005.md) handles (e.g. `!id.HasValue || id == 0`) — DL3005 takes precedence
- The `!` expression is inside an `is true`/`is false` pattern that [DL3003](DL3003.md) handles (e.g. `(!id.HasValue) is true`) — DL3003 takes precedence

## Examples

### Violation

```csharp
if (!flag) { }                    // DL3004
if (!string.IsNullOrEmpty(s)) { } // DL3004
if (!(o is string)) { }          // DL3004
if (!(o is null)) { }            // DL3004
```

### Fixed

```csharp
if (flag is false) { }
if (string.IsNullOrEmpty(s) is false) { }
if (o is not string) { }
if (o is not null) { }
```

### No diagnostic

```csharp
// Already using pattern matching
if (flag is false) { }
if (o is not null) { }

// Nullable bool — different semantics
bool? nullable = null;
var result = !nullable;

// De Morgan's — different concern
if (!(a && b)) { }
```

## Code fix

For `!(x is pattern)`, replaces with `x is not pattern`. For `!(x is T)` (old-style type check), replaces with `x is not T`. For all other boolean negations, replaces `!expr` with `expr is false`.
