# DL3018 - Namespace does not match folder structure

Flags namespace declarations that do not correspond to the file's location in the project directory tree.

| Rule   | Category | Default Severity | Code Fix |
| ------ | -------- | ---------------- | -------- |
| DL3018 | Style    | Warning          | Yes      |

## Why

When namespaces drift from the folder structure, navigating the codebase becomes harder — you can no longer rely on the file system to locate types. Keeping namespaces aligned with directories is a widely adopted convention (enforced by default in Rider/ReSharper) that makes the project predictable and reduces friction when searching, refactoring, or onboarding.

## When the diagnostic fires

The expected namespace is computed as `RootNamespace` + directory segments from the project root to the file. A diagnostic is reported when the declared namespace differs from this expected value.

- File at project root with wrong namespace: `namespace Wrong;` when expected `MyProject`
- File in a subfolder with wrong namespace: `namespace MyProject;` in `Models/Foo.cs` when expected `MyProject.Models`
- File in a deep subfolder with wrong namespace: `namespace MyProject;` in `Models/Entities/Foo.cs` when expected `MyProject.Models.Entities`

## When the diagnostic does NOT fire

- The declared namespace matches the expected namespace based on folder structure
- The file has no namespace declaration (top-level statements, global usings)
- Nested (inner) namespace declarations — only the outermost namespace is checked
- The file path is outside the project directory
- `build_property.ProjectDir` is not available (e.g. ad-hoc scripts)
- Generated code (controlled by `GeneratedCodeAnalysisFlags.None`)

## Examples

### Violation

Given a project with `RootNamespace = MyProject` and the following file at `Models/User.cs`:

```csharp
namespace MyProject; // DL3018: expected MyProject.Models

public class User { }
```

Or a completely wrong namespace:

```csharp
namespace SomethingElse; // DL3018: expected MyProject.Models

public class User { }
```

### Fixed

```csharp
namespace MyProject.Models;

public class User { }
```

### No diagnostic

```csharp
// File at project root
namespace MyProject;

public class Program { }
```

```csharp
// File at Services/OrderService.cs
namespace MyProject.Services;

public class OrderService { }
```

## Overlap with IDE0130

Microsoft ships [IDE0130](https://learn.microsoft.com/en-us/dotnet/fundamentals/code-analysis/style-rules/ide0130), a built-in code-style rule that performs the same check. DL3018 exists because IDE0130 has practical limitations that reduce its usefulness in CI pipelines:

1. **Disabled by default on `dotnet build`.** All `IDExxxx` rules are code-style rules, and the .NET SDK disables code-style analysis during command-line builds. To enable IDE0130 in CI the consuming project must set `<EnforceCodeStyleInBuild>true</EnforceCodeStyleInBuild>` **and** add `<CompilerVisibleProperty>` entries for `RootNamespace` and `ProjectDir`.
2. **Extra consumer configuration.** DL3018 ships the required `CompilerVisibleProperty` entries in its [`.props` file](../../src/DemoriLabs.CodeAnalysis/buildTransitive/DemoriLabs.CodeAnalysis.props), so it works out of the box during `dotnet build` with zero additional setup.
3. **Known reliability issues.** IDE0130 has a history of false positives and incorrect messages across .NET SDK versions (see [dotnet/roslyn#73261](https://github.com/dotnet/roslyn/issues/73261), [#76484](https://github.com/dotnet/roslyn/issues/76484), [#55014](https://github.com/dotnet/roslyn/issues/55014)).

If you already have `EnforceCodeStyleInBuild` enabled and prefer IDE0130, disable DL3018:

```ini
dotnet_diagnostic.DL3018.severity = none
```

## Code fix

Replaces the declared namespace with the expected namespace derived from the folder structure. The fix preserves all surrounding trivia (whitespace, comments) and works with both file-scoped and block-scoped namespace declarations.

## Configuration

The analyzer reads two MSBuild properties exposed via `AnalyzerConfigOptionsProvider.GlobalOptions`:

- **`build_property.RootNamespace`** — the root namespace for the project. Falls back to the assembly name if not set.
- **`build_property.ProjectDir`** — the project directory. Required; the analyzer is silently skipped if this is unavailable.

Both properties are automatically provided by the .NET SDK during build; no manual configuration is needed.

```ini
# Adjust severity
dotnet_diagnostic.DL3018.severity = error

# Disable entirely
dotnet_diagnostic.DL3018.severity = none
```
